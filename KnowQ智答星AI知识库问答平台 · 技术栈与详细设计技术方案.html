服务职责：

  • 管理服务：知识库CRUD、问答系统配置、LLM配置、用户权限管理。
  • 问答引擎：接收前端提问，根据问答系统配置执行 AI模式 或 规则模式，返回结果。
  • 统计服务：消费问答日志，聚合统计数据，提供看板API。
  • Celery Workers：处理文档导入、向量化等异步任务，解耦核心请求链路。

三、核心模块详细设计

3.1 知识库管理模块

3.1.1 数据模型（核心表）

-- 知识库
CREATE TABLE knowledge_bases (
    id UUID PRIMARY KEY,
    name VARCHAR(200) NOT NULL,
    description TEXT,
    created_by UUID REFERENCES users(id),
    status VARCHAR(20) DEFAULT 'active', -- active/archived
    created_at TIMESTAMPTZ DEFAULT now()
);

-- 知识条目（支持问答对和文档块）
CREATE TABLE knowledge_entries (
    id UUID PRIMARY KEY,
    kb_id UUID NOT NULL REFERENCES knowledge_bases(id) ON DELETE CASCADE,
    type VARCHAR(10) NOT NULL CHECK (type IN ('q_a', 'chunk')),
    question VARCHAR(500),           -- 用于问答对
    content TEXT NOT NULL,           -- 答案或段落文本
    source_doc VARCHAR(500),         -- 来源文档名
    chunk_index INT,                 -- 文档块序号
    embedding vector(1536),          -- OpenAI text-embedding-ada-002 维度，或按需调整
    created_at TIMESTAMPTZ DEFAULT now()
);

CREATE INDEX idx_entries_kb_id ON knowledge_entries(kb_id);
CREATE INDEX idx_entries_embedding ON knowledge_entries USING ivfflat (embedding vector_cosine_ops);

3.1.2 文档导入流程（异步）

  1. 管理员上传文件 → 管理服务接收，存储至对象存储（MinIO/S3），返回任务ID。
  2. 发送消息到 Redis 队列，Celery Worker 消费：
    • 使用 Unstructured 库解析文档（支持PDF/Word/Markdown/TXT），提取文本并分块（默认512 tokens，重叠50 tokens）。
    • 对每个分块调用 Embedding 服务（可复用 LLM 配置中的 embedding 模型）生成向量。
    • 批量写入 knowledge_entries 表（type='chunk'）。
  3. 完成后更新导入记录状态，管理平台可查看进度。

3.1.3 关键字检索（BM25）

在规则模式下，不依赖向量。为支持快速关键字匹配，可在 knowledge_entries 表上建立全文索引，或使用 Elasticsearch。

  • 方案A（轻量）：使用 PostgreSQL 的 tsvector 进行全文搜索，结合 plainto_tsquery。
  • 方案B（强扩展）：同步数据至 Elasticsearch，利用其强大的相关性评分和同义词过滤器。

3.2 问答引擎设计（核心）

问答引擎根据问答系统的配置（qa_systems 表中的 mode 和 kb_ids）执行不同流程。

3.2.1 AI 模式处理流程

用户问题 question
    │
    ▼
[1] 提取系统配置: 关联知识库 IDS, LLM配置, 是否启用联网搜索
    │
    ▼
[2] 向量检索知识库 (top K, 如 K=5)
    │
    ├─ 使用 question embedding 在 pgvector 中搜索余弦相似度最高的条目
    │
    ▼
[3] 构建 Prompt 并调用 LLM 进行意图识别与判断
    │  System Prompt 示例:
    │  "你是一个智能助手。首先判断用户问题是否与提供的知识库内容相关。
    │   如果相关，请严格基于知识库内容回答；如果不相关，回答'__USE_SEARCH__'。"
    │  Context: [检索到的知识条目文本]
    │  User: question
    │
    ▼
[4] 解析 LLM 回复:
    ├── 若回复包含 "__USE_SEARCH__" 或空相关内容 → 启用联网搜索 (如果配置开启)
    │       └─ 调用 Tavily API 搜索 question，获取搜索结果摘要
    │       └─ 调用 LLM 二次生成: 结合搜索结果和原始问题生成自然语言回复，标注来源
    └── 否则直接使用 LLM 基于知识库生成的答案，标注“基于知识库生成”

关键优化：

  • 避免两次 LLM 调用（判断+生成），可结合 function calling：让 LLM 决定是否需要搜索，并返回结构化字段。
  • 向量检索前可先进行关键词匹配过滤，减少无效上下文。

3.2.2 规则模式处理流程

用户问题 question
    │
    ▼
[1] 预处理: 分词 (jieba/IK Analyzer)、同义词扩展、拼写纠正
    │
    ▼
[2] 检索知识库:
    ├─ 方式1: PostgreSQL full-text search (tsvector) 根据相关性排序
    ├─ 方式2: 精确匹配问答对中的 question 字段 (编辑距离/文本相似度)
    │
    ▼
[3] 判断匹配度:
    ├─ 最高匹配得分 >= 阈值 (如 0.7) → 取出最佳匹配 content，生成回复
    └─ 否则 → 返回预设提示：“未找到相关内容，请尝试其它关键词”

算法细节：可集成 BM25 或 TF-IDF 进行排序，并利用 pg_trgm 扩展进行模糊匹配。

3.4 联网搜索集成

  • 封装 TavilySearchClient，使用异步 HTTP 调用。
  • 支持参数化配置：max_results，include_answer，search_depth。
  • 结果缓存：对相同搜索词在短期内（如5分钟）缓存结果，减少费用和延迟。

3.5 问答前端系统设计

  • 动态路由：每个问答系统有一个唯一子域名或路径，如 https://qa.example.com/{system_id}。
  • 前端实现：
    • 利用 Next.js 的 getStaticPaths（如系统数量不多）或客户端动态获取配置。
    • 页面加载时调用 GET /api/system/{id}/config 获取系统名称、logo等。
    • 聊天组件采用流式接收：请求 POST /api/ask/{id} 并设置 Accept: text/event-stream。
  • 管理平台中“生成问答前端”：实质上是在数据库创建配置记录，并返回一个URL，无需实际构建前端文件。

3.6 管理平台详细设计

  • 前端页面：
    • 知识库列表/编辑器：支持富文本编辑问答对，拖拽上传文档。
    • 问答系统创建向导：分步选择知识库、模式、LLM配置、搜索配置。
    • LLM 配置页：下拉列表内置13个选项，选择后展示对应字段；自定义选项提供开放表单。
    • 统计看板：使用 ECharts/AntV 展示图表，数据通过统计服务 API 获取。
  • 权限控制：基于 JWT，前端路由守卫，后端中间件验证角色。

四、数据统计与用户管理

4.1 问答日志表（补充索引）

CREATE TABLE qa_logs (
    id UUID PRIMARY KEY,
    system_id UUID NOT NULL,
    user_id UUID,
    question TEXT NOT NULL,
    answer TEXT,
    mode VARCHAR(10),          -- 'ai' or 'rule'
    source VARCHAR(10),        -- 'kb', 'web', 'rule_match', 'rule_none'
    latency_ms INT,
    feedback SMALLINT,         -- 1: up, -1: down, 0: none
    created_at TIMESTAMPTZ DEFAULT now()
);
-- 建立时间、system_id 索引优化统计查询

4.2 统计服务

  • 定时任务（Celery Beat）每小时聚合 qa_logs 数据到统计汇总表，或直接查询原表（小规模可行）。
  • 提供 REST API：GET /stats/dashboard?system_id=&range=30d

4.3 用户管理

  • 用户表角色字段 role 区分 admin/user。
  • 管理员可创建普通用户；普通用户无法登录管理平台。
  • 问答前端可无需登录（根据需求），或支持简单的访问密码保护（由管理员为问答系统设置密码，前端要求输入）。

五、安全与性能设计

层面措施
通信安全全站 HTTPS，Nginx 启用 HSTS
密钥管理使用 Fernet/AES 加密存储 API Key，运行时解密，不允许通过日志打印
速率限制基于 IP/User 的令牌桶算法，防止滥用
输入消毒防止 prompt injection，对用户输入做长度限制和敏感词过滤
性能- 向量检索使用 IVF+PQ 索引加速
- Redis 缓存热点知识库条目
- 异步任务分离文档处理
高可用Kubernetes 多副本部署，数据库主从复制+自动故障转移，Redis Sentinel

六、部署与运维

6.1 部署拓扑（K8s）

  • Namespace: qa-system
  • Deployments: admin-api, qa-engine, stats-service, celery-workers
  • Ingress: 负责路由 /admin/ 到 admin-api，/ask/ 到 qa-engine，前端静态资源通过 CDN 或 Nginx 容器提供。
  • 配置文件通过 ConfigMap/Secrets 管理。

6.2 CI/CD 流程

  1. 代码提交 → 运行单元测试、lint
  2. 构建 Docker 镜像 → 推送到私有仓库
  3. 使用 ArgoCD 监听 Helm Chart 变更，自动同步到 K8s 集群
  4. 生产环境采用蓝绿或滚动发布策略，确保零停机。
