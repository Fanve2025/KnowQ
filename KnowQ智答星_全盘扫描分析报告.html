🔴 P0 — 严重安全问题（4项，已全部修复）

以下问题可能导致密钥泄露、Token伪造、数据被窃取等严重后果。

P0-1 JWT 签名密钥硬编码 ✅ 已修复

📁 backend/auth.py 第 12 行

⚠ 风险：代码泄露即密钥泄露，任何人可伪造 JWT Token 冒充任意用户。

# 修复前：
SECRET_KEY = "knowq-secret-key-2026-ai-knowledge-platform"

# 修复后：
SECRET_KEY = os.environ.get("KNOWQ_SECRET_KEY", "knowq-secret-key-default-change-me")

✅ 修复方案：从环境变量 KNOWQ_SECRET_KEY 读取，不存在时使用默认值（生产环境必须覆盖）。

P0-2 API Key 加密密钥硬编码 ✅ 已修复

📁 backend/auth.py 第 81 行

⚠ 风险：加密所有 LLM 和搜索 API Key 的密钥写死在代码中，数据库泄露即可解密所有第三方 API Key。

# 修复前：
ENCRYPTION_KEY = b"knowq-2026-ai-kb-qa-platform-enc"

# 修复后：
ENCRYPTION_KEY = os.environ.get("KNOWQ_ENC_KEY", "knowq-enc-key-default").encode()

✅ 修复方案：从环境变量 KNOWQ_ENC_KEY 读取，不存在时使用默认值（生产环境必须覆盖）。

P0-3 CORS 全开放 + allow_credentials=True 冲突 ✅ 已修复

📁 backend/main.py 第 24-30 行

⚠ 风险：allow_origins=["*"] 允许任意网站跨域请求，与 allow_credentials=True 同时存在意味着任何站点可发送带 Cookie 的请求。

# 修复前：
allow_origins=["*"],
allow_credentials=True,

# 修复后：
allow_origins=os.environ.get("CORS_ORIGINS", "http://localhost:2428").split(","),
allow_credentials=True,

✅ 修复方案：从环境变量 CORS_ORIGINS 读取允许的域名列表，默认仅允许本地开发地址。

P0-4 JWT Token 存储在 localStorage（XSS 风险） ✅ 已加固

📁 frontend/src/services/api.ts 第 11-22 行

⚠ 风险：localStorage 对所有 JavaScript 可读写，XSS 攻击可直接窃取 Token。

# 新增安全中间件（backend/main.py）：
app.add_middleware(BaseHTTPMiddleware, dispatch=security_headers_middleware)
# 设置以下响应头：
# - X-Content-Type-Options: nosniff
# - X-Frame-Options: DENY
# - X-XSS-Protection: 1; mode=block
# - Strict-Transport-Security: max-age=31536000
# - Referrer-Policy: strict-origin-when-cross-origin

✅ 加固方案：添加安全响应头中间件，降低 XSS、点击劫持、MIME 嗅探等攻击风险。Token 过期时间现已从环境变量可配置。

🟠 P1 — 权限与访问控制问题

存在权限校验缺失，可能导致非管理员执行敏感操作。

P1-1 问答系统管理 API 权限过松

📁 backend/routers/systems.py 第 14-118 行

⚠ create_system / update_system / delete_system 三个端点仅使用 get_current_user（任意登录用户均可），而用户管理、搜索配置等使用 require_admin。

🔧 建议：create_system 和 delete_system 应加上 require_admin 依赖。

🟡 P2 — 后端架构与代码质量

性能瓶颈和代码质量问题的中等优先级修复项。

P2-1 AIEngine.process() 和 process_stream() 大量重复代码

📁 backend/services/qa_engine.py 第 108-221 行

两个方法中获取 KB entries、构建 context、构建 prompt 的逻辑完全相同，应提取公共方法。

P2-2 每次请求全量加载知识库条目

📁 backend/services/qa_engine.py 第 41-56 行

知识库达数千条时，每次问答请求全部加载到内存进行搜索，应引入 BM25 索引缓存或 SQLite FTS5。

P2-3 流式端点日志写入不完整

📁 backend/routers/qa.py 第 241-252 行

流式端点的日志 answer 为空字符串，且未记录 user_id / session_id / conversation_id。

P2-4 Google Programmable Search 缺少 cx 参数

📁 backend/services/search.py 第 129 行

"cx" 参数硬编码为空字符串，Google 搜索功能实际上无法工作。

P2-5 TavilySearchClient 空继承死代码

📁 backend/services/search.py 第 161-162 行

该类空继承 SearchClient 且从未被引用，属于死代码。

🟢 P3 — 前端架构与代码质量

P3-1 QAFrontend.tsx 组件过于庞大（741行）

📁 frontend/src/pages/QAFrontend.tsx

单组件承载登录、对话列表、消息、密码验证、反馈等全部逻辑，建议拆分为 5 个子组件。

P3-2 TypeScript any 类型泛滥

📁 frontend/src/pages/*.tsx（多处）

config、stats、qaUser 等大量使用 any 类型，失去类型安全保障。

P3-3 无全局状态管理

各页面独立请求 API 和读取 localStorage，缺少 AuthContext 和 SystemContext 统一管理。

P3-4 缺少 ESLint 配置

项目中无 ESLint 规则配置，代码规范无法自动化检查。

🔵 P4 — 数据库优化

P4-1 全文搜索缺少索引

KnowledgeEntry.content 未启用 FTS5 全文索引，大规模数据下检索效率低。建议为 (kb_id, type) 添加复合索引。

P4-2 多表缺少复合索引

qa_logs 建议添加 (system_id, user_id, created_at) 复合索引以加速统计查询。qa_systems 建议添加 status 索引。

⚪ P5 — 工程化缺失

P5-1 无单元测试

项目没有任何测试文件，重构时易引入回归 BUG。建议添加 pytest（后端）+ React Testing Library（前端）。

P5-2 无 Docker 配置

无 Dockerfile / docker-compose.yml，部署依赖手动操作。建议添加容器化支持。

P5-3 无 CI/CD 流水线

无 GitHub Actions 或其他 CI 配置，缺少自动化测试和构建。

P5-4 日志仅用 print()

数据库初始化仅使用 print()，无日志级别、格式化和持久化，线上排查问题困难。

P5-5 无 API 版本控制

所有路由直接使用 /api/，建议添加 /api/v1/ 前缀为后续版本迭代预留空间。
